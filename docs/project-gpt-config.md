## 1. 项目概览

| 项目 | 内容 |
|---|---|
| 项目名称 | 家脉亲缘 |
| 核心方向 | 家庭亲属关系、家族结构、亲缘确认、家族文化 |
| 产品形态 | 后端、后台管理端、官网 PC/H5、微信小程序 |
| 设计风格 | 温馨、传统、稳重、清晰 |
| 当前重点 | 重写 web 端亲属关系工具 |
| 重要原则 | 不再继续零散补丁式修改，优先重构完整文件 |

### 项目目标

家脉亲缘是一个围绕家庭关系和亲属结构的项目，主要目标包括：

- 管理家族、成员、亲属关系。
- 展示公开关系树。
- 提供亲属称谓推导工具。
- 支持官网、H5、小程序和后台管理。
- 保持温馨传统的视觉风格，避免夸张娱乐化设计。

## 5. 启动方式

### 后端

~~~powershell

~~~

### 后台管理端

~~~powershell

~~~

### 官网 web

~~~powershell

~~~

### 一键启动

~~~powershell
cd D:\GitHub\zhangyulei\Tree
.\start_all.bat
~~~

### 常见问题

| 问题 | 原因 | 处理 |
|---|---|---|
| Vite proxy ECONNREFUSED | 后端没有启动 | 先启动 backend |
| localhost 能访问，手机不能访问 | 服务只监听本机或防火墙限制 | 检查监听地址、防火墙、局域网 IP |
| 页面空白 | 前端编译报错或运行时报错 | 看终端红色错误和浏览器 Console |

## 6. Git 规范

### 应该提交

| 类型 | 示例 |
|---|---|
| 后端源码 | backend |
| 后台源码 | admin |
| 官网源码 | web |
| 小程序源码 | miniprogram |
| 文档 | docs |
| 依赖声明 | package.json、go.mod |
| 项目说明 | README.md |
| 必要脚本 | start_all.bat |

### 不应该提交

| 类型 | 示例 |
|---|---|
| 依赖目录 | node_modules |
| 构建产物 | dist |
| 环境变量 | .env、.env.local |
| 临时脚本 | fix_*.cjs、*.ps1 |
| 备份文件 | *.bak、*.bak_* |
| 临时垃圾目录 | .trash_* |
| 报错截图 | 临时调试截图 |

### 建议 .gitignore

~~~gitignore
node_modules/
dist/
.env
.env.local
*.log
*.bak
*.bak_*
*.tmp
*.ps1
*.cjs
.trash_*/
.DS_Store
.vscode/settings.json
~~~

## 7. 后台管理端规则

后台管理端主要功能：

| 功能 | 说明 |
|---|---|
| 登录页 | 管理员登录 |
| 首页看板 | 后台概览 |
| 官网配置 | 管理官网基础信息 |
| 轮播图管理 | 管理首页轮播 |
| 文章管理 | 家族故事、项目内容 |
| 留言管理 | 官网留言 |
| 公开树展示 | 管理官网公开关系树 |

### 菜单结构要求

家族、成员、关系应该嵌套在一个菜单中，不要全部并列。

~~~text
家族管理
├─ 家族列表
├─ 成员管理
└─ 亲属关系
~~~

家族较多时，需要支持：

- 搜索
- 分页
- 状态筛选
- 快速进入成员管理
- 快速进入亲属关系管理

## 8. 官网和公开关系树规则

### 官网风格

| 要求 | 说明 |
|---|---|
| 温馨 | 符合家庭、亲情、传承主题 |
| 传统 | 不做过度潮流和娱乐化 |
| 清晰 | 页面层级明确，信息易读 |
| 稳重 | 适合作为正式官网 |

### 公开关系树

公开关系树只用于官网展示，不能读取用户真实家庭树数据。

| 数据类型 | 是否可用于官网公开展示 |
|---|---|
| 后台配置的公开树数据 | 可以 |
| 用户真实家庭树数据 | 不可以 |
| 小程序用户私人数据 | 不可以 |

### 示例人物

公开关系树可以参考历史人物关系，例如：

| 人物 | 说明 |
|---|---|
| 汉武帝刘彻 | 中心人物 |
| 汉景帝刘启 | 父亲 |
| 王娡 | 母亲 |
| 田蚡 | 舅舅 |
| 陈阿娇 | 皇后 |
| 卫子夫 | 皇后 |
| 刘据 | 子女 |
| 刘弗陵 | 子女 |

### 展示要求

- PC 端可以展示完整关系网。
- H5 端尽量一屏展示。
- 中心人物明确。
- 其他人物卡片更小，避免重叠。
- 关系线清晰。
- 关系标签最好显示在线上。
- 页面风格保持温馨传统。
- 不使用热搜风、八卦风、夸张娱乐风。

## 9. 编码和乱码处理规则

### 必须遵守

| 规则 | 说明 |
|---|---|
| 使用 UTF-8 | 所有源码和文档必须 UTF-8 |
| 使用 VS Code | 不使用 Windows 记事本 |
| 不写乱码脚本 | PowerShell 中不要写大量中文源码 |
| 不基于乱码文件继续改 | 发现乱码应恢复干净版本 |
| 不生成大量临时文件 | 避免 ps1、cjs、bak 堆积 |

### VS Code 建议配置

文件路径：

~~~text
.vscode/settings.json
~~~

内容：

~~~json
{
  "files.encoding": "utf8",
  "files.autoGuessEncoding": false,
  "files.eol": "\n"
}
~~~

## 10. GPT 协作要求

### 应该做

- 修改前说明改哪些文件。
- 修改后说明改了什么。
- 报错时先看文件和行号。
- 复杂功能优先重构完整文件。
- 保留已经正确的亲属关系规则。
- 文档和源码都使用 UTF-8。
- 亲属关系工具规则以 docs/kinship-tool-spec.md 为准。

### 不应该做

- 不要反复打补丁。
- 不要瞎猜函数名。
- 不要生成一堆临时脚本。
- 不要把 web 工具问题扩散到 backend/admin/miniprogram。
- 不要为了修一个小问题破坏已经能跑的功能。
